📐 レイアウト設計

画面を上2/3と下1/3に分割。
上2/3: 4ブースの入力カード（各ブースは横1行＝コード+IN+OUT+残+補）
下1/3: キーボード非表示時に集計パネルが見える。キーボード表示時は自然に隠れる。

UX: 入力中は入力に集中、手を止めたら数字が見える

⌨️ 入力フロー

Enter順: B01-IN → B01-OUT → B01-残 → B01-補 → B02-IN → ...
4ブース×4フィールド = 16回のEnterで1機械完了（現状は40回）。
設定値A〜OやPrize名は「変更がある時だけ」タップで展開。

UX: 入力回数60%削減

📊 集計パネルのデータ

機械小計（4ブース合計）:
① 今回差額 — 今入力したIN差分×単価の合計
② 前回差額 — 前回入力時のIN差分合計
③ 累計差額 — 最後の集金からの累計売上
④ 月間予測 — 今月の日割り×残り日数
⑤ 前月実績 — 先月の合計売上

店舗合計: 上と同じ5項目を全機械で合算。

パフォーマンス: 下記参照

⚡ パフォーマンス見積もり

必要なAPI呼び出し（1機械表示時）:
① booths（4行）— 既存キャッシュ有
② meter_readings（最新2件×4ブース）— 既存
③ 店舗合計用の全ブース最新readings — ★新規

③が重い？→ 2つの対策:
A) 初回ロード時に店舗全体のサマリーを1回取得してキャッシュ
→ meter_readings!A:Z を全取得（~数百行）、JSでフィルタ
→ Google Sheets API は1回のGETで十分。レスポンス 0.3〜0.8秒程度

B) booth_monthly_summary シート（月次集計テーブル）を活用
→ 集計済みデータを読むだけなので軽い。ただし集計バッチが必要

結論: A案で十分実用的。170台÷4ブース=680行のmeter_readingsでも、 最新2件×680=1360行。JSONで約200KB。初回1秒以内でキャッシュ後は即表示。

結論: 重くない。初回1秒キャッシュでOK

🔄 データフロー

sheets.js に追加する関数:
getStoreReadingsSummary(storeId) — 店舗全体のreadings取得→集計
getCollectionBaseline(storeId) — 最終集金日のメーター値
getMonthlyTotal(storeId, month) — 月間合計（前月実績用）

キャッシュ戦略: 店舗切替時に1回フェッチ → メモリキャッシュ → 入力保存時にインクリメンタル更新
