# Team Review & Fix ワークフロー

対象: $ARGUMENTS

## 手順

### Step 1: 変更差分の取得
```bash
git diff --name-only HEAD~1  # または指定されたコミット範囲
git diff HEAD~1              # 全差分
```

### Step 2: 並列レビュー
変更ファイルの内容に応じて、以下の観点でSubagentを並列起動する：

#### Agent A: ロジックレビュー
- バグ・ロジックエラーがないか
- エッジケースの見落としがないか
- エラーハンドリングが適切か（ただし過剰でないか）

#### Agent B: 一貫性レビュー
- 既存コードのパターンに沿っているか
- 命名規則が統一されているか
- 不要なコード（デバッグログ、コメントアウト等）がないか

#### Agent C: セキュリティ・パフォーマンスレビュー
- XSS, インジェクション等の脆弱性がないか
- 不要なAPIコール・再レンダリングがないか
- メモリリークの懸念がないか

### Step 3: 問題の集約
各Subagentのレビュー結果を集約し、重要度で分類する：
- **CRITICAL**: 必ず修正が必要（バグ、セキュリティ脆弱性）
- **WARNING**: 修正を推奨（パフォーマンス、可読性）
- **INFO**: 軽微な改善提案

### Step 4: 自動修正
- CRITICAL と WARNING の問題を自動で修正する
- 各修正の内容を記録する
- INFOはユーザーに報告のみ

### Step 5: 検証
1. `npm run build` でビルド確認
2. 修正前後の差分を確認し、意図しない変更がないことを検証

### Step 6: 報告
以下のフォーマットで結果を報告する：

```markdown
## Review & Fix 結果

### 発見された問題
| # | 重要度 | ファイル | 問題 | 対応 |
|---|--------|---------|------|------|
| 1 | CRITICAL | ... | ... | 修正済 |
| 2 | WARNING | ... | ... | 修正済 |
| 3 | INFO | ... | ... | 報告のみ |

### 修正内容
- ...

### ビルド結果: PASS / FAIL
```

完了通知を送る。
